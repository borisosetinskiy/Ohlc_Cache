# Архитектура OHLC Cache

## Обзор системы

OHLC Cache - это высокопроизводительная система кэширования временных рядов финансовых данных, построенная на принципах off-heap хранения и оптимизированной индексации.

## Основные принципы проектирования

### 1. Off-heap хранение
- Использование Chronicle Map для хранения данных вне кучи JVM
- Снижение нагрузки на Garbage Collector
- Увеличение доступной памяти для данных
- Прямой доступ к памяти для максимальной производительности

### 2. Чанкование данных
- Разделение временных рядов на логические блоки (чанки)
- Оптимизация для чтения и записи больших объемов данных
- Эффективное использование памяти

### 3. Асинхронная индексация
- Отдельный поток для обновления индексов
- Неблокирующие операции записи
- Высокая пропускная способность

## Компоненты системы

### 1. Модели данных

#### Candle Interface
```java
public interface Candle extends Byteable, BytesMarshallable {
    int CANDLE_STRUCT_SIZE = 52; // Размер структуры в байтах
    
    @Group(1) double getHigh();
    @Group(2) double getLow();
    @Group(3) double getOpen();
    @Group(4) double getClose();
    @Group(5) double getVolume();
    @Group(6) long getTime();
    @Group(7) int getTick();
}
```

**Особенности:**
- Использование Chronicle Values для оптимизации памяти
- Фиксированный размер структуры (52 байта)
- Поддержка flyweight паттерна

#### Chart Model
Контейнер для запросов исторических данных:
- Метаданные запроса (символ, период, временной диапазон)
- Результаты поиска
- Статистика выполнения

### 2. Система кэширования

#### CandleTimeSeriesCache
Основной компонент для работы с временными рядами:

**Ключевые возможности:**
- Вставка одиночных свечей
- Пакетная вставка массивов свечей
- Поиск по временным диапазонам
- Управление чанками данных

**Структура чанка:**
```
[Header: 40 bytes] + [Candles: N * 52 bytes]
Header:
- Capacity (4 bytes): количество свечей в чанке
- Count (4 bytes): текущее количество свечей
- FirstTimestamp (8 bytes): время первой свечи
- LastTimestamp (8 bytes): время последней свечи
- CreatedTimestamp (8 bytes): время создания чанка
- LastAccessTimestamp (8 bytes): время последнего доступа
```

#### ChronicleIndexApi
API для работы с индексами:

**Функции:**
- Асинхронное обновление индексов
- Поиск чанков по временным диапазонам
- Группировка индексов по временным интервалам

**Структура индекса:**
```
IndexKey:
- SeriesId (String): идентификатор серии
- IndexGroupTimestamp (long): временная метка группы

IndexValue:
- Массив временных меток чанков
- Оптимизированное хранение для быстрого поиска
```

## Алгоритмы и оптимизации

### 1. Бинарный поиск в чанках
```java
public int binarySearch(long timestamp) {
    int low = 0, high = getCount() - 1;
    while (low <= high) {
        int mid = (low + high) >>> 1;
        long midTs = page.readLong(HEADER_SIZE + (long) mid * CANDLE_SIZE + 40);
        if (midTs < timestamp) low = mid + 1;
        else if (midTs > timestamp) high = mid - 1;
        else return mid;
    }
    return -(low + 1);
}
```

### 2. Управление памятью
- Использование `Bytes.allocateDirect()` для off-heap памяти
- Автоматическое освобождение ресурсов через `AutoCloseable`
- Пул объектов для переиспользования

### 3. Потокобезопасность
- Использование `ExternalMapQueryContext` для атомарных операций
- Блокировки на уровне серий для пакетных операций
- Неблокирующие очереди для индексации

## Потоки данных

### 1. Вставка данных

```
Входящие данные → Валидация → Определение чанка → 
Блокировка → Вставка в чанк → Обновление индекса → Разблокировка
```

### 2. Чтение данных

```
Запрос → Поиск в индексе → Загрузка чанков → 
Фильтрация по времени → Возврат результатов
```

### 3. Индексация

```
Новые чанки → Очередь индексации → Асинхронная обработка → 
Группировка по интервалам → Обновление индексов
```

## Производительность

### Ожидаемые характеристики

- **Запись:** 100,000+ свечей/сек
- **Чтение:** 1,000,000+ свечей/сек
- **Память:** ~52 байта на свечу
- **Латентность:** <1 мс для одиночных операций

### Факторы производительности

1. **Размер чанка:** Оптимальный баланс между памятью и производительностью
2. **Размер индекса:** Влияет на скорость поиска
3. **Конфигурация Chronicle Map:** Количество сегментов, размер страниц
4. **Системные ресурсы:** RAM, CPU, дисковая подсистема
